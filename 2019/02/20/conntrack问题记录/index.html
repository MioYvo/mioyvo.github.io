conntrack问题记录

发表于 2019-02-20

记录一次nf_conntrack引发的服务器异常

起因

突然发现v2ray用不了了，登录服务器总是失败，多次尝试终于登上去，发现ping和docker pull都报operation not permitted，但用户是root，百思不得其解。

查看syslog发现大量报nf_conntrack:table full, dropping packet，询问得知前几天服务器被攻击过，于是上网找办法，查了一圈，现在已解决，总结并记录一下。

参考

  • https://testerhome.com/topics/15824
  • https://blog.csdn.net/dog250/article/details/9373715
  • https://security.stackexchange.com/questions/43205/nf-conntrack-table-full-dropping-packet

前情提要

内核中有个conntrack模块，会跟踪所有网络连接，记录存在一个哈希表中，如果服务器比较繁忙，新连接进来的速度远高于释放的速度，把哈希表塞满了，新连接的数据包就会被丢掉。如果有人 DDoS 攻击的话情况更糟，无论是空连接攻击还是简单地用短连接发大量请求都能轻易塞满哈希表。

检查

  • 查看netfilter 相关内核参数

    sysctl -a | grep conntrack

    旧配置：

    • nf_conntrack_max 的默认值算法为

      CONNTRACK_MAX = RAMSIZE (in bytes) / 16384 / (ARCH / 32)
      其中 ARCH 为 CPU 架构，值为 32 或 64。
      这台机器的内存只有481M，481*1024*1024/16384/(64/32)=15392

    • nf_conntrack_buckets 默认值算法为

      HASHSIZE = CONNTRACK_MAX / 4

    • net.netfilter.nf_conntrack_tcp_timeout_established：432000 （5天，默认值）

      • 这个值对应的场景是 “双方建立了连接后一直不发包，直到 5 天后才发” ……
      • 查询过sysctl -a | grep net.ipv4，keep-alive 超时时间只有 18 小时多一点（net.ipv4.tcp_keepalive_time + net.ipv4.tcp_keepalive_intvl * net.ipv4.tcp_keepalive_probes），由于超时关 socket 不发包，conntrack 无法根据包头的标识知道状态的变化，记录会一直处于 ESTABLISHED 状态，直到 5 天后倒计时结束才删掉。
      • 空连接攻击的最佳目标。攻击者把 IP 包头的源地址改成随机 IP，握完手就关 socket，用一台机发请求就能把你的哈希表填满。

    • net.netfilter.nf_conntrack_tcp_timeout_syn_recv：60 （默认）

      类似，故意不发握手的 ACK 即可。但这个超时时间没那么夸张，系统也有 syn cookie 机制来缓解 syn flood 攻击。

    • net.netfilter.nf_conntrack_tcp_timeout_syn_sent：120 （默认）

      • 程序的连接超时时间，120秒对于普通服务器来说太长了
    • net.netfilter.nf_conntrack_tcp_timeout_fin_wait: 120 （默认）
      • net.ipv4.tcp_fin_timeout 默认 60 秒
    • net.netfilter.nf_conntrack_icmp_timeout: 30 （默认）
      • ping超时时间，30秒太长了

解决

解决方法最粗暴的是直接关掉防火墙或者在iptables里设置不跟踪某些协议的连接，但这在经常遭受攻击的生产服务器中显然是不现实的。但直接加大nf_conntrack的最大数量和哈希表大小会增大内存的使用影响服务。故调整conntrack参数，来尽量避免攻击带来的影响。

参考 某路由器公司的配置

改过后观察了一下，主程序和v2ray都没有问题，写入配置文件/etc/sysctl.d/90-conntrack.conf随机器启动

应用文件中的配置
sysctl -p /etc/sysctl.d/90-conntrack.conf

总结

平常觉得Linux的默认参数配置是够用的，其实不然，有些参数因为各种原因配置得并不合理，需要根据使用环境进行自行配置。
